# Engine 部分答辩讲稿：结构设计与仿真环境搭建

## 1. 我的分工定位

我负责的是项目中的 `Engine` 部分，主要工作是搭建整个系统的底层仿真环境和结构设计。

简单来说，Engine 的作用不是实现某一个单独的调度算法，而是提供一个统一、可复现、可扩展的实验环境。它负责把城市路网、车辆、任务、充电站、调度策略和日志系统组织起来，使不同算法可以在相同规则下运行和比较。

可以在答辩中这样介绍：

> 我的分工是 Engine 模块，也就是整个项目的仿真内核。它负责建立城市物流调度问题的基础环境，包括图结构路网、车辆状态、任务生命周期、电量消耗、充电排队以及统一的调度接口。后续的启发式算法、强化学习策略、离线优化方法和前端可视化，都是基于这个 Engine 来运行的。

Engine 在整个项目中的位置可以概括为：

```text
地图与任务数据
    ↓
Engine 仿真环境
    ↓
调度策略 / 强化学习 / 离线优化
    ↓
实验日志 / 前端可视化 / 论文分析
```

所以我这部分的核心目标是：让项目有一个稳定的“实验平台”。

## 2. Engine 总体架构

Engine 目录主要分为两个部分：

```text
Engine/
├── Framework/          核心仿真框架
└── Map Resource/       地图数据与预处理资源
```

其中 `Framework` 是主要代码目录：

```text
Framework/
├── core/               核心仿真内核
├── scheduler/          调度策略接口和基线算法
├── generator/          地图、任务、站点生成器
├── api/                FastAPI 后端接口
├── configs/            实验配置文件
└── examples/           可运行实验入口
```

各模块职责如下：

| 模块 | 作用 |
| --- | --- |
| `core` | 定义图结构、车辆、任务、充电站、路径规划和仿真主循环 |
| `scheduler` | 定义统一调度器接口，并实现最近任务、最早截止期、最大重量等基线策略 |
| `generator` | 构造随机地图、真实地图、任务集合和车辆集合 |
| `api` | 提供后端服务，使前端可以实时获取仿真状态 |
| `configs` | 用 YAML 管理不同实验场景和参数 |
| `examples` | 提供命令行实验入口，方便复现实验结果 |

答辩时可以强调：

> 我在设计时采用了分层结构。`core` 只负责仿真规则和状态转移，`scheduler` 只负责根据当前状态选择动作，`generator` 负责构造场景，`logger` 负责记录结果。这样可以降低模块之间的耦合，后续替换地图、替换调度算法或接入前端时，不需要大规模修改核心代码。

## 3. 核心建模一：城市路网图结构

城市物流调度的基础是路网，因此 Engine 首先把城市道路抽象成带权图。

在图结构中：

- 节点表示仓库、路口、任务点或充电站。
- 边表示道路连接关系。
- 边权表示距离或通行成本。
- 车辆移动和调度决策都基于图上的最短路径。

可以这样讲：

> 城市道路天然适合用图结构建模。我们没有简单使用二维欧氏距离，而是把道路抽象成带权图，通过图搜索算法计算车辆到任务点、任务点到仓库、车辆到充电站之间的路径。这样模型更贴近真实物流场景，也方便后续接入真实地图。

路径规划部分由 `PathFinder` 负责，主要能力包括：

- 计算两个节点之间的最短距离。
- 返回最短路径上的节点序列。
- 判断车辆当前电量是否足够完成任务并返回仓库。
- 为车辆选择合适的充电站。

这一部分可以突出数据结构和算法基础：

> 路网是图结构，路径规划依赖最短路算法。调度器做决策时并不是只看任务编号，而是通过图上的距离和可达性判断任务是否可执行。

## 4. 核心建模二：实体对象设计

Engine 中的核心实体包括：

```text
Vehicle             车辆
Task                配送任务
ChargingStation     充电站
Depot               仓库
```

### 4.1 Vehicle 车辆模型

车辆模型记录了车辆在仿真中的全部动态信息，包括：

- 当前所在节点。
- 电池容量和当前电量。
- 最大载重。
- 当前状态。
- 当前任务。
- 当前路径。
- 累计行驶距离。
- 累计得分。

车辆状态采用状态机设计：

```text
IDLE
  ↓
MOVING_TO_TASK
  ↓
MOVING_TO_DEPOT

低电量时：

IDLE / MOVING
  ↓
MOVING_TO_CHARGE
  ↓
WAITING_CHARGE
  ↓
CHARGING
  ↓
IDLE
```

答辩时可以说：

> 我把车辆行为设计成状态机。车辆每个时间步都会根据当前状态进行转移，比如空闲、前往任务、返回仓库、前往充电站、排队等待充电、正在充电等。这样仿真逻辑比较清晰，也便于后续增加新的车辆行为。

### 4.2 Task 任务模型

任务模型主要包含：

- `release_time`：任务释放时间。
- `deadline`：任务截止时间。
- `origin_node`：任务所在节点。
- `weight`：任务重量。
- `status`：任务状态。

任务生命周期如下：

```text
FUTURE → PENDING → ASSIGNED → COMPLETED
                    ↓
                 EXPIRED
```

解释：

- `FUTURE`：任务还没有到达释放时间。
- `PENDING`：任务已释放，等待调度。
- `ASSIGNED`：任务已经分配给车辆。
- `COMPLETED`：车辆到达任务点，任务完成。
- `EXPIRED`：超过截止时间仍未完成。

可以这样讲：

> 任务不是一开始全部出现，而是按照释放时间动态进入系统。这模拟了真实即时配送场景中订单不断到来的情况。调度器只能看到当前已经释放的任务，因此这个问题是动态在线调度问题。

### 4.3 ChargingStation 充电站模型

充电站模型包含：

- 所在节点。
- 充电桩数量。
- 当前占用的充电桩。
- 等待队列。
- 充电速率。

充电流程如下：

```text
车辆电量不足
    ↓
选择充电站
    ↓
移动到充电站
    ↓
有空闲桩：开始充电
无空闲桩：进入等待队列
    ↓
充电完成后恢复空闲状态
```

可以这样讲：

> 新能源物流调度和普通车辆调度最大的区别之一是电量约束。车辆不能无限行驶，而且充电站也不是无限资源。因此我在仿真环境中加入了充电桩占用和排队机制，使调度策略需要同时考虑任务收益、路径距离和充电压力。

## 5. 核心建模三：仿真环境 Environment

`Environment` 是 Engine 的核心类，负责维护整个仿真世界的状态。

它包含：

- 当前时间。
- 图结构路网。
- 所有车辆。
- 所有任务。
- 所有充电站。
- 仓库节点。
- 当前待处理任务集合。
- 已分配任务集合。
- 已完成任务集合。
- 已超时任务集合。
- 总得分。
- 日志系统。

每一个仿真时间步的流程如下：

```text
1. 释放当前时间到达的新任务
2. 检查任务是否超时
3. 自动处理需要协同配送的任务
4. 调用调度器选择动作
5. 执行动作并检查合法性
6. 更新车辆移动和电量消耗
7. 更新充电站排队和充电状态
8. 记录当前时间步日志
9. 时间向前推进
```

答辩时可以这样讲：

> Environment 是一个离散时间仿真器。每经过一个时间步，它会先释放新任务，然后检查超时任务，再让调度器根据当前状态选择车辆和任务的匹配，之后更新车辆位置、电量、任务完成情况和充电站状态。最后记录日志并推进时间。

这部分是答辩重点，因为它直接体现你搭建了完整的仿真环境。

## 6. 调度接口设计

为了让不同算法可以接入同一个仿真环境，我设计了统一的调度器接口。

调度器只需要实现一个核心方法：

```text
select_actions(env)
```

这个方法输入当前环境状态，输出调度动作。

动作可以是：

```text
(vehicle_id, task_id)
```

表示把某辆车派给某个任务。

也可以是协同任务动作：

```text
{
  "task_id": task_id,
  "vehicle_allocations": {
    vehicle_id: load
  }
}
```

表示多个车辆共同完成一个任务。

答辩时可以这样讲：

> 调度器本身不直接修改环境状态，它只提出动作。真正的合法性检查和状态更新都由 Environment 完成。这样可以保证不同策略在同一套规则下运行，也避免每个策略重复实现车辆移动、电量判断和任务状态更新逻辑。

目前 Engine 中实现了几种基线调度策略：

| 策略 | 思路 |
| --- | --- |
| 最近任务优先 | 空闲车辆选择距离最近的可行任务 |
| 最早截止期优先 | 优先处理 deadline 最早的任务 |
| 最大重量优先 | 优先处理重量最大的任务 |

这些策略可以作为后续强化学习和离线优化方法的比较基线。

## 7. 电量约束与可行性检查

电量约束是 Engine 设计中的重点。

派单时不是简单判断车辆是否空闲，还要检查：

```text
当前车辆电量
  是否足够从当前位置到任务点
  是否足够完成任务后返回仓库
  是否满足安全电量余量
```

如果不满足，车辆不会被分配该任务，而是会被引导去充电。

可以这样讲：

> 这个设计避免了车辆在路上因为电量不足而无法完成任务的情况。调度器提出动作后，Environment 会检查路径和电量可行性。如果不可行，这个动作不会被执行，车辆会优先进入充电流程。

充电站选择也不只考虑距离，还会考虑站点负载，例如：

- 队列长度。
- 已占用充电桩数量。
- 从当前位置到充电站的距离。

这使得仿真更接近真实新能源汽车调度问题。

## 8. 动态任务与在线调度

Engine 中的任务不是静态全部给定，而是根据 `release_time` 动态释放。

这样有三个好处：

1. 更接近真实订单不断到达的场景。
2. 调度器不能提前知道所有未来任务，体现在线决策特点。
3. 可以观察不同策略在动态压力下的表现。

可以这样讲：

> 在每个时间步，Environment 会检查是否有新任务到达。如果任务释放时间小于等于当前时间，就把它从未来任务集合移动到待调度任务集合。调度器只能选择当前已经释放的任务，因此整个问题是动态在线调度问题。

任务超时机制也很重要：

```text
如果 current_time > deadline 且任务未完成
    任务状态变为 EXPIRED
    从待调度或已分配集合移除
    计入失败任务并产生惩罚
```

这让调度策略必须考虑任务紧急程度，而不是只考虑距离。

## 9. 协同任务设计

Engine 中还支持协同任务扩展，也就是一个任务可以由多辆车共同完成。

设计思路：

- 普通任务由一辆车完成。
- 如果任务重量超过单车容量，可以拆分给多辆车。
- 每辆车承担一部分载重。
- 任务根据累计送达重量判断是否完成。

可以这样讲：

> 为了让系统具备扩展性，我在任务模型中保留了协同配送能力。当一个任务重量超过单车容量时，可以将任务拆分给多辆车，每辆车承担一部分载重，最后根据累计完成量判断任务是否完成。

如果答辩时间紧，可以只把这一点作为“扩展能力”简单带过。

## 10. 日志系统与实验复现

Engine 不只输出最终结果，还会记录完整仿真过程。

主要日志包括：

| 日志 | 内容 |
| --- | --- |
| `step_log` | 每个时间步的整体指标 |
| `vehicle_log` | 车辆位置、电量、状态、里程 |
| `task_log` | 任务释放、分配、完成、超时 |
| `station_log` | 充电站排队和占用 |
| `events` | 关键事件记录 |

日志可以导出为：

```text
JSON
CSV
```

答辩时可以说：

> 日志系统的作用是保证实验可复现、可分析。我们不仅知道某个策略最终得分是多少，还可以回看每个时间步发生了什么，分析车辆为什么去充电、任务为什么超时、某个策略为什么表现更好。

这些日志后续可以用于：

- 绘制实验图表。
- 对比不同调度策略。
- 分析充电策略影响。
- 支撑论文中的实验结论。
- 给前端提供可视化数据。

## 11. 地图和场景生成

Engine 支持两类场景：

```text
随机地图场景
真实地图场景
```

随机地图场景的作用：

- 快速调试。
- 批量实验。
- 控制变量。
- 生成 small / medium / large 不同规模问题。

真实地图场景的作用：

- 使用番禺区域地图数据。
- 将真实道路转成图结构。
- 提高实验的现实意义。

可以这样讲：

> 为了兼顾调试效率和真实感，Engine 同时支持随机地图和真实地图。两种地图最后都会转换成统一的 Graph 结构，因此上层调度器和仿真逻辑不需要关心地图来源。

这一点体现了抽象设计：

```text
随机地图     ┐
            ├── 统一 Graph 接口 ── Environment
真实地图     ┘
```

## 12. 后端接口与前端可视化支持

Engine 中的 `api` 模块使用 FastAPI 提供后端接口。

主要作用：

- 启动仿真实例。
- 暂停、继续、停止仿真。
- 获取当前仿真状态。
- 通过 WebSocket 推送实时状态。
- 为前端地图可视化提供数据。

答辩时可以说：

> Engine 不只是命令行实验工具，也可以作为后端服务运行。前端通过 API 获取车辆位置、任务状态、充电站状态和统计指标，从而实现实时可视化。

这样讲可以说明你的 Engine 是系统级模块，而不只是脚本集合。

## 13. 可以重点强调的设计亮点

答辩最后可以总结 5 个亮点：

### 13.1 图结构建模

城市道路被抽象为带权图，车辆路径规划和可达性判断都基于图算法。

### 13.2 状态机驱动仿真

车辆、任务和充电站都有明确状态，使仿真过程清晰稳定。

### 13.3 调度策略解耦

Environment 和 Scheduler 分离，不同算法可以通过统一接口接入。

### 13.4 新能源车辆约束

考虑车辆电量、充电站排队、充电桩占用和安全电量余量。

### 13.5 实验可复现

完整记录 JSON/CSV 日志，支持实验统计、论文图表和前端可视化。

可以用这段作为总结：

> 总体来说，我的工作重点是搭建一个统一、可扩展、可复现的仿真平台。它将城市路网、动态任务、车辆电量、充电约束和调度策略统一组织起来，使不同算法可以在同一个环境下公平比较，也为后续实验分析和前端展示提供基础。

建议你的 Engine 部分控制在 6 到 8 页。

### 第 1 页：我的分工

标题：Engine 仿真环境与结构设计

内容：

- 负责底层仿真内核。
- 建立路网、车辆、任务、充电站模型。
- 提供调度接口和日志输出。
- 支撑算法实验和前端可视化。

讲法：

> 我负责的是项目的 Engine 部分，它是整个系统的底层运行环境。后续所有策略都依赖这个环境进行状态更新、动作执行和结果记录。

### 第 2 页：整体架构

放结构图：

```text
Generator → Core Environment → Scheduler → Logger/API
```

讲法：

> Engine 采用分层设计，生成器负责构造场景，核心环境负责仿真，调度器负责决策，日志和 API 负责输出结果。

### 第 3 页：核心数据结构

内容：

- Graph：城市路网。
- Vehicle：车辆状态。
- Task：任务生命周期。
- ChargingStation：充电站队列。

讲法：

> 城市道路用图结构表示，车辆和任务用状态机管理，充电站引入队列和桩位占用，构成完整的物流调度环境。

### 第 4 页：仿真主循环

放流程：

```text
释放任务 → 检查超时 → 调度决策 → 更新车辆 → 更新充电站 → 记录日志
```

讲法：

> 每个时间步，Environment 都会按照这个顺序推进仿真。这样可以模拟动态任务到达、车辆移动、电量变化和充电排队。

### 第 5 页：调度接口设计

内容：

- Scheduler 只负责选择动作。
- Environment 负责合法性检查和执行。
- 支持最近任务、最早截止期、最大重量、强化学习等策略。

讲法：

> 调度策略和仿真环境是解耦的，因此我们可以方便地替换策略，并保证不同策略在同一规则下比较。

### 第 6 页：电量与充电约束

内容：

- 派单前检查能否完成任务并返回仓库。
- 低电量车辆自动前往充电站。
- 充电站有排队和桩位限制。

讲法：

> 新能源物流调度的关键难点是电量和充电资源有限，因此 Engine 专门建模了电量消耗和充电站排队。

### 第 7 页：日志与实验支持

内容：

- step log。
- vehicle log。
- task log。
- station log。
- events。

讲法：

> 日志系统保证实验结果可复现，也为后续论文图表和前端可视化提供数据来源。

### 第 8 页：总结

内容：

- 统一仿真平台。
- 可扩展策略接口。
- 支持随机地图和真实地图。
- 支持新能源车辆约束。
- 支持实验复现和可视化。

讲法：

> 我的工作为整个项目提供了底层环境，使算法部分、实验部分和展示部分能够连接起来。

## 15. 可能被问到的问题与回答

### Q1：为什么要单独设计 Engine，而不是直接写调度算法？

答：

> 因为调度算法需要在统一的环境下比较。如果每个算法都自己处理车辆移动、电量消耗和任务状态，结果就不公平，也难以维护。所以我先搭建 Engine，把仿真规则统一起来，算法只需要通过接口输出动作。

### Q2：为什么使用图结构建模道路？

答：

> 城市道路本身就是由路口和道路组成的网络，非常适合抽象成图。使用图结构后，可以用最短路径算法计算车辆到任务、到仓库、到充电站的距离，也方便接入真实地图。

### Q3：调度器怎么和环境交互？

答：

> 调度器通过 `select_actions(env)` 获取当前环境状态，并返回车辆和任务的匹配动作。Environment 会检查动作是否合法，包括车辆是否空闲、任务是否待处理、载重是否足够、电量是否可行，然后再执行状态更新。

### Q4：如何避免车辆电量不足？

答：

> 派单前会做可行性检查，判断车辆是否能从当前位置到任务点，并在完成任务后返回仓库，同时保留安全电量余量。如果不满足条件，车辆不会接这个任务，而是优先前往充电站。

### Q5：充电站是怎么模拟的？

答：

> 每个充电站有固定数量的充电桩。如果车辆到达时有空闲桩，就开始充电；如果没有空闲桩，就进入等待队列。每个时间步会更新车辆电量，充满后释放充电桩，并让队列中的下一辆车进入充电。

### Q6：为什么说这个环境支持在线调度？

答：

> 因为任务不是一开始全部给出，而是根据释放时间动态进入系统。调度器只能处理当前已经释放的任务，不能直接使用未来任务信息，所以这是动态在线调度环境。

### Q7：Engine 如何支持强化学习？

答：

> 强化学习把 Engine 当作环境。每次智能体选择一个调度规则或动作，Engine 执行这个动作并返回新的状态和奖励。也就是说，Engine 负责状态转移和结果评估，强化学习负责决策。

### Q8：日志有什么用？

答：

> 日志可以记录每个时间步的车辆、任务、充电站和事件信息。这样不仅能得到最终分数，还能分析策略行为，比如哪些任务超时、车辆什么时候充电、充电站是否拥堵，从而支持实验分析和论文图表。

### Q9：随机地图和真实地图如何统一？

答：

> 随机地图和真实地图最终都会转换为统一的 Graph 结构。Environment 和 Scheduler 只依赖 Graph 接口，不关心地图来源，因此可以在不同场景下复用同一套仿真逻辑。

### Q10：你这部分最大的难点是什么？

答：

> 最大的难点是把多个约束统一在一个稳定的仿真流程中，包括动态任务、车辆状态、电量消耗、充电排队、任务超时和调度动作合法性。需要保证每个时间步的状态更新顺序清晰，否则很容易出现任务重复分配、车辆状态不一致或电量判断错误。

## 17. 三分钟完整讲稿

如果你有三分钟左右，可以用这一版：

> 我负责的是项目中的 Engine 部分，主要是结构设计和仿真环境搭建。Engine 是整个系统的底层运行平台，它不直接代表某一个具体算法，而是为所有调度策略提供统一的环境。后续的启发式算法、强化学习策略、离线优化方法和前端可视化，都是基于这个 Engine 来运行的。
>
> 在整体结构上，我把 Engine 分成了几个模块。`core` 是核心仿真内核，负责图结构、实体对象、路径规划和仿真主循环；`scheduler` 定义统一调度接口，并实现最近任务、最早截止期、最大重量等基线策略；`generator` 用来构造随机地图、真实地图、车辆和任务；`api` 提供后端接口，支持前端实时展示；`configs` 用来管理不同实验场景的参数。
>
> 在建模方面，城市路网被抽象为带权图。节点表示仓库、路口、任务点或充电站，边表示道路，边权表示距离。车辆路径规划和任务可达性检查都基于图上的最短路径。这样做的好处是，随机地图和真实地图最后都能统一成 Graph 结构，上层算法不需要关心地图来源。
>
> 车辆、任务和充电站都采用状态化建模。车辆有空闲、前往任务、返回仓库、前往充电站、等待充电、正在充电等状态；任务有未释放、待调度、已分配、已完成和已超时状态；充电站则维护充电桩占用情况和等待队列。通过状态机方式推进仿真，可以让每个时间步的逻辑更清晰。
>
> Environment 是 Engine 的核心。每个时间步中，它会先释放到达时间的新任务，再检查任务是否超时，然后调用调度器选择动作。动作执行前，Environment 会检查车辆是否空闲、载重是否足够、电量是否能完成任务并返回仓库。如果动作合法，就更新车辆路径和任务状态。之后系统继续更新车辆移动、电量消耗、充电站排队和充电进度，最后记录日志并推进时间。
>
> 我还设计了统一调度接口。调度器只需要实现 `select_actions(env)`，根据当前环境状态返回动作。这样不同策略可以很方便地接入同一个仿真环境，并且所有策略都由 Environment 执行同样的合法性检查和状态更新，保证实验比较公平。
>
> 这个 Engine 的一个重点是新能源车辆约束。车辆不能无限行驶，所以派单前要检查电量可行性；如果电量不足，车辆会前往充电站。充电站也不是无限资源，而是有固定充电桩和等待队列。因此调度策略需要同时考虑任务距离、截止时间、电量和充电拥堵。
>
> 最后，Engine 会完整输出 step、vehicle、task、station 和 event 日志。这些日志既可以用于实验复现和论文图表，也可以通过后端接口提供给前端做实时可视化。因此我的工作可以总结为：搭建了一个统一、可扩展、可复现的新能源物流调度仿真平台，为整个项目的算法实验和展示提供底层支撑。
