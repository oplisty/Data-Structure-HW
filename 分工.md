 算法与建模 op：逻辑大脑
这是项目的核心，负责处理最复杂的调度策略和数学模型。
基础任务： 实现“最近任务优先 (Greedy)”和“最大任务优先 (Longest/Heaviest First)”等启发式基准算法。
进阶任务（难度分）： 构建 静态全局最优模型（上帝视角），编写线性规划方程，调用 Gurobi 或 CPLEX。
实现一种 元启发式算法（如遗传算法 GA 或大邻域搜索 ALNS）来处理动态任务的重新规划。
环境模拟： 编写任务生成器（包含时间轴、坐标、重量、电量消耗逻辑）。



B. 系统架构与数据结构组 xgy：基础设施
负责将物理世界的规则转化为代码逻辑。
地图实现： 使用图结构（Adjacency List/Matrix）存储道路、充电站、仓库和任务点。
寻路引擎： 实现 Dijkstra 或 A* 算法，并需考虑“电量约束下的路径搜索”。
对象抽象： 定义 Vehicle（电量、载重、当前路径）、Task（产生时间、截止时间、重量）、ChargingStation（排队队列、充电效率）等类。



C. 仿真可视化与文档组 xyx：成果展示
决定了“主观分”和“附加分”的关键。
图形界面 (GUI)： 使用 Python (PyQt/PyGame) 或前端技术展示车辆在地图上的实时移动、电量变化和任务完成状态。
数据统计： 自动生成对比图表（如：启发式 vs. Gurobi 的得分对比、里程对比、排队时间分析）。
最终交付： 录制演示视频，撰写包含算法流程图、 数学公式推导和结果分析的项目报告。

### 更推荐的职责划分

**A. 算法与策略组**
负责：

* 调度策略接口设计
* 最近任务优先、最大任务优先
* 可选进阶：GA / ALNS / RL / 静态最优模型
* 策略评估指标设计（得分、超时率、总里程、平均等待时间等）

不建议 A 单独负责环境模拟器底层实现，最多负责提出规则。

**B. 仿真内核与数据结构组**
负责：

* 地图图结构
* 路径搜索（Dijkstra / A*）
* 电量消耗、载重约束、充电逻辑
* 任务生成器、时间推进器、事件系统
* Vehicle / Task / ChargingStation / Depot 等类
* 给 A 组提供统一 API，比如：

  * `get_available_tasks(t)`
  * `get_vehicle_state(i)`
  * `dispatch(vehicle_id, task_id)`
  * `step()` / `run()`

这一组其实最像“引擎组”。

**C. 可视化与实验分析组**
负责：

* GUI 动画展示
* 仿真日志可视化
* 自动统计图表
* 报告、流程图、视频
* 配合 A 组做实验对比展示

这样三个人依赖关系会更清晰：
**B 先搭引擎，A 在引擎上写策略，C 读取日志做展示。**

### 你们当前分工里，最应该补的一点

你们需要提前约定一个 **统一数据流**，不然最后会很乱。建议统一成：

* B 组输出：仿真环境 `Environment`
* A 组输入：`Environment` 当前状态，输出调度决策
* C 组输入：每个时间步的日志 `log.json/csv`

例如日志里至少有：

* 时刻
* 车辆位置
* 当前电量
* 当前载重
* 正在执行的任务
* 任务完成/超时状态
* 充电站排队长度
* 累计得分

这样 C 组不用碰核心代码，也能独立做图和 GUI。

### 从课程作业角度看，你们这样安排能覆盖要求吗

能，前提是最低目标先收缩一下。你们最好分成“必做版”和“加分版”。

**必做版**

* 动态任务生成
* 至少两种调度策略
* 至少三种规模
* 电量不足时去充电
* 充电站排队
* 结果统计与简单可视化

**加分版**

* 多车协同任务
* 元启发式 / 强化学习
* 上帝视角 + Gurobi/CPLEX 对比
* 完整 GUI 动画

因为三人小组最怕一开始目标开太满，最后每块都半成品。
就这类课程项目来说，**一个稳定可运行的基线系统 + 两种策略对比 + 清晰可视化**，通常比“想做 RL + Gurobi + GUI，但最后都没跑通”更容易拿高分。
